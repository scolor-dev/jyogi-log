# ADR 0003 — リフレッシュトークンをローテーション方式にする

| 項目 | 内容 |
|------|------|
| ステータス | 決定済み |
| 決定日 | 2026-03-01 |
| 関連テーブル | `refresh_tokens` |

---

## 背景

ADR 0002 にて RefreshToken を Opaque Token として DB 管理することを決定した。  
次に「リフレッシュトークンをどう更新するか」、特に **トークンの使い回しを許容するかどうか** を検討する必要があった。

---

## 問題

リフレッシュトークンが漏洩した場合、攻撃者はそのトークンを使い続けてアクセストークンを取得し続けることができる。  
漏洩を検知する仕組みがなければ、ユーザーが気づかない間にセッションを乗っ取られるリスクがある。

---

## 検討した選択肢

### A. 使い回し方式（再利用あり）— 却下

- リフレッシュトークンは固定で、何度でも使ってアクセストークンを再発行できる
- 実装がシンプル
- トークンが漏洩しても検知する手段がない
- 漏洩したトークンは有効期限が切れるまで悪用され続ける

### B. ローテーション方式（再利用なし）— 採用

- リフレッシュトークンを使うたびに新しいトークンを発行し、古いトークンは即時失効させる
- 失効済みのトークンが再度使われた場合、**トークン再利用攻撃（token reuse attack）** として検知できる
- 検知時はそのユーザーの全トークンを一括失効し、再ログインを強制することでセッション乗っ取りを防ぐ
- 実装コストはAより高いが、セキュリティ上の優位性が大きい

---

## 決定

**選択肢 B（ローテーション方式）を採用する。**

```
リフレッシュ（正常系）
  → 受け取った token_hash を refresh_tokens で照合
  → 旧トークンの revoked_at = NOW()（即時失効）
  → 新 RefreshToken を生成し INSERT
  → 新 AccessToken（JWT）を発行して返す

リフレッシュ（再利用攻撃検知）
  → 受け取った token_hash が既に revoked_at を持つ場合
  → そのトークンに紐づく全 refresh_tokens を一括失効
  → 401 を返しクライアントに再ログインを要求
```

`refresh_tokens` テーブルは失効済みレコードも削除せず `revoked_at` で保持することで、再利用の検知に使用する。

---

## 影響と制約

- クライアントはリフレッシュのたびに新しいトークンを保存し直す必要がある
- ネットワーク障害等でレスポンスが届かなかった場合、クライアントが古いトークンで再試行すると再利用攻撃として誤検知される可能性がある。この問題は現時点では許容し、必要であれば将来的に猶予ウィンドウの導入を検討する
- 失効済みの `refresh_tokens` レコードは再利用検知のために一定期間保持し、その後クリーンアップバッチで削除する
