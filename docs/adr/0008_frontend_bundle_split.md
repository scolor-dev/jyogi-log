# ADR 0008 — ログイン画面をアプリ本体と分割しない（単一バンドル）

| 項目 | 内容 |
|------|------|
| ステータス | 決定済み |
| 決定日 | 2026-03-12 |
| 関連 | Frontend / Vite バンドル設定 |

---

## 背景

React は SPA として単一の JS バンドルを生成し、クライアントへ送信する。  
ログイン画面とアプリ本体（ダッシュボード等）を同一バンドルにまとめると、未認証ユーザーがログイン画面を開いた時点でアプリ本体のコードも一緒にダウンロードされる。

この挙動が問題になりうるかを検討した。

---

## 問題の候補

- 未認証ユーザーにダッシュボード等のコードが送信される
- ログイン画面の初期ロード時に不要なコードのダウンロード・パース・コンパイルが発生する

---

## 検討した選択肢

### A. 単一バンドル（分割しない）— 採用

- Vite のデフォルト設定のまま、認証ページとアプリ本体を単一バンドルとして配信する
- 実装・ビルド設定がシンプルに保てる
- 不要なコードが送信されるが、**対象ユーザーが限定されているため実用上の問題にならない**

### B. 認証ページ群 vs アプリ本体の2分割 — 却下

- Vite の `build.rollupOptions` でエントリーポイントを分割し、ログイン・サインアップ等の認証ページを別チャンクにする
- 未認証ユーザーへの送信データ量を削減できる
- ビルド設定・ルーティングが複雑になる
- **ユーザーが限定されているため、パフォーマンス改善の恩恵が小さくコストが見合わない**

---

## 決定

**選択肢 A（単一バンドル）を採用する。**

本システムのユーザーは限定されており、バンドルサイズの最適化による恩恵よりも実装のシンプルさを優先する。  
将来的にユーザー規模が拡大した場合や、バンドルサイズが問題になった場合は Route-based code splitting への移行を検討する。

---

## 影響と制約

- 将来ページ数・機能が増えバンドルサイズが大きくなった場合は、`React.lazy` + `Suspense` による Route-based code splitting への移行を検討する
- 現時点ではバンドルサイズの計測・監視は行わない
