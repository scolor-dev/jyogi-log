# ADR 0000 — 技術スタックの選定（Rust / React / PostgreSQL）

| 項目 | 内容 |
|------|------|
| ステータス | 決定済み |
| 決定日 | 2026-02-16 |

---

## 背景

プロジェクト開始にあたり、バックエンド・フロントエンド・データベースの技術スタックを選定する必要があった。

---

## バックエンド — Rust

### 検討した選択肢

| 言語 / FW | 特徴 |
|-----------|------|
| **Rust** | メモリ安全性・高パフォーマンス・強力な型システム |
| Go | シンプル・学習コスト低・並行処理が得意 |
| Node.js (TypeScript) | フロントエンドと言語統一できる・エコシステムが豊富 |
| Python (FastAPI) | 学習コスト低・プロトタイプに向く |

### 決定: Rust を採用

- **認証基盤という性質上、脆弱性リスクを言語レベルで下げたかった**: バッファオーバーフロー・use-after-free・メモリリーク等のセキュリティ脆弱性のクラスを Rust は原理的に排除する。パスワードハッシュ・トークン管理を扱う基盤として、言語レベルの安全性保証は強い選定根拠になる
- **メモリ安全性・パフォーマンス**: GC がなく低レイテンシが期待できる。コンパイル時にメモリ関連のバグ（バッファオーバーフロー・use-after-free 等）を排除する。セキュリティが重要な認証基盤において信頼性が高い
- **型安全性・堅牢性**: 強力な型システムとコンパイラによる厳格なチェックにより、実行時エラーをコンパイル時に検出できる。`Option` / `Result` による明示的なエラーハンドリングが実装の品質を高める

### 却下した理由（他選択肢）
- **Go**: シンプルさは魅力だが、型安全性・メモリ管理の厳格さで Rust に劣る。null 安全でなく、データ競合もランタイムまで検出されない場合がある
- **Node.js**: フロントエンドとの言語統一は魅力だが、シングルスレッドモデルでデータ競合は少ないが、メモリ安全性の保証がなく型安全性も TypeScript 止まり。（実行時には剥がれる）型安全性・堅牢性で Rust に劣る
- **Python**: プロトタイプには向くが、動的型付けで実行時エラーが多く、認証基盤の堅牢性要件を満たしにくい。本番の認証基盤には型安全性が不十分

---

## フロントエンド — React

### 検討した選択肢

| FW / ライブラリ | 特徴 |
|----------------|------|
| **React** | エコシステムが最大規模・SPA 構築に向く・学習リソースが豊富 |
| Vue.js | 学習コスト低・テンプレート構文が直感的 |
| Svelte | ビルド時コンパイル・軽量 |
| Next.js | SSR/SSG 対応・React ベース |

### 決定: React を採用

- **チームの既存スキル**: チームが React の経験を持っており、立ち上げコストを抑えられる
- **エコシステムの豊富さ**: React Router DOM・Zustand・Tailwind CSS 等、認証 UI の構築に必要なライブラリが揃っており、選択肢が広い

### 却下した理由（他選択肢）

- **Vue.js / Svelte**: チームスキルが React に寄っているため、切り替えコストが見合わない
- **Next.js**: SSR が現時点の要件では不要。シンプルな SPA に留める

---

## データベース — PostgreSQL

### 検討した選択肢

| DB | 特徴 |
|----|------|
| **PostgreSQL** | ACID 保証・JSONB・部分インデックス・豊富な型 |
| MySQL | 普及率が高い・シンプル |
| SQLite | 組み込み・ファイルベース・開発向け |

### 決定: PostgreSQL を採用

- **トランザクション・ACID 保証**: 認証基盤はトークン発行・失効・ローテーション等、整合性が崩れると即セキュリティ問題に直結する操作が多い。ACID 保証が厚い PostgreSQL が適切
- **JSON/JSONB 対応**: `user_credentials.secret_meta` のように構造が可変なデータを型安全に扱える。JSONB はインデックスも張れるため検索性能も確保できる

また、部分インデックス（`WHERE revoked_at IS NULL` 等）や `pgcrypto` 拡張など、認証基盤の設計で活用できる機能が豊富な点も採用を後押しした。

### 却下した理由（他選択肢）

- **MySQL**: 部分インデックスが使えない（本システムの設計で多用している）・JSONB のサポートが劣る
- **SQLite**: 並行書き込みに弱く、本番の認証基盤には不適切

---

## まとめ

| レイヤー | 選定技術 | 主な理由 |
|---------|---------|---------|
| バックエンド | Rust | メモリ安全性・パフォーマンス・型安全性・堅牢性 |
| フロントエンド | React | チームの既存スキル・エコシステムの豊富さ |
| データベース | PostgreSQL | ACID 保証・JSONB 対応 |
