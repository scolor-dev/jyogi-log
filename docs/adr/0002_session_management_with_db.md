# ADR 0002 — AccessToken を DB で持たない（JWT によるステートレス認証）

| 項目 | 内容 |
|------|------|
| ステータス | 決定済み |
| 決定日 | 2026-03-01 |
| 関連テーブル | `refresh_tokens` |

> **注記**: この ADR はトークン方式の選定を記録したものである。  
> その後 `sessions` テーブルを導入する設計変更が行われた。経緯は [ADR 0003](./0003_introduce_sessions_table.md) を参照。

---

## 背景

認証方式として JWT（JSON Web Token）を採用することは決定していた。  
問題は JWT をどう運用するか、特に **「ログアウト・強制失効をどう実現するか」** という点だった。

JWT はそれ自体に有効期限と署名を持つため、サーバー側に状態を持たない（ステートレス）設計が可能だが、それには以下のトレードオフがある。

---

## 問題

JWT のみでステートレスに認証を管理すると、発行済みトークンをサーバー側から無効化する手段がない。

- ユーザーがログアウトしても、アクセストークンの有効期限が切れるまで当該トークンは有効なまま残る
- アカウント停止・パスワード変更・不正アクセス検知などで即時失効させたい場面で対応できない
- リフレッシュトークンの再利用攻撃（token reuse attack）を検知・対処する仕組みが作れない

---

## 検討した選択肢

### A. セッション・クッキー方式 — 却下

- 多くの Web フレームワークで標準的にサポートされており、ログアウトや強制失効が確実に行える
- しかしサーバー側でセッション情報を保持する必要があり、メモリ消費が増大する
- Restful API との親和性が低く、モバイルアプリ等のクロスプラットフォーム対応がしづらい
- スケーラビリティの観点から採用しない

### B. Opaque Token 方式 — AccessToken としては却下、RefreshToken として採用

- クロスプラットフォームでの実装が容易で、モバイルアプリなどでも扱いやすい
- トークンに紐づく情報の管理が自由で、即時失効が容易
- **AccessToken として使う場合**: 毎リクエストで DB アクセスが必要になり、DB への負荷が高くスケーラビリティが低下するため却下
- **RefreshToken として使う場合**: リフレッシュは頻度が低いため DB アクセスのコストが許容範囲。ローテーションや再利用攻撃の検知も DB があることで実現できる → **採用**

### C. JWT — AccessToken として採用

- ステートレスで、署名検証のみで認証が完結する（DB アクセス不要）
- 改ざん検知が可能
- トークン自体の無効化は原理的にできないが、有効期限を短く設定（15分程度）することでリスクを許容範囲に抑える
- **AccessToken として採用する**

---

## 決定

**AccessToken と RefreshToken でトークン方式を使い分ける。**

| トークン | 方式 | 保存場所 | 有効期限 |
|---------|------|---------|---------|
| AccessToken | JWT（選択肢C） | サーバー側 DB なし | 短期（15分程度） |
| RefreshToken | Opaque Token（選択肢B） | `refresh_tokens.token_hash` | 長期（30日程度） |

```
ログイン
  → RefreshToken（ランダム文字列）を生成し token_hash を refresh_tokens に INSERT
  → AccessToken（JWT）を発行（payload: sub=user_uuid, exp）

通常リクエスト（保護エンドポイント）
  → AccessToken の JWT 署名検証のみ（DB アクセスなし）

ログアウト
  → 該当 refresh_tokens.revoked_at = NOW()

リフレッシュ
  → refresh_tokens の token_hash を照合（revoked_at IS NULL かつ expires_at > NOW()）
  → 旧トークンを revoked_at で失効（ローテーション）
  → 新 RefreshToken を INSERT、新 AccessToken（JWT）を発行
  → 失効済みトークンが使われた場合はそのユーザーの全トークンを失効（再利用攻撃対策）

強制失効（アカウント停止・不正検知等）
  → user_id に紐づく refresh_tokens を一括失効
  → 既発行の AccessToken は exp まで有効（遅延を許容）
```

---

## 影響と制約

- 通常リクエストは JWT 検証のみのため、強制失効は AccessToken の有効期限（15分程度）まで遅延する。これは意図的なトレードオフとして許容する
- 期限切れの `refresh_tokens` レコードは定期的にクリーンアップするバッチが必要
- 将来的に AccessToken の即時失効が必要になった場合、Redis ブロックリスト + `jti` 照合への移行を妨げない設計にする
